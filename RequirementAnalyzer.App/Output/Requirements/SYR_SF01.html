  • 7.11.2.2.1 Description
  • 7.11.2.2.1.0-1 Brief description: CCU is mixed ASIL multi core system, hence to ensure the Freedom from Interference software partitioning is mandatory between software components with different ASIL software. To achieve freedom from interference between applications /tasks / ISRs on the same core access rights of these runtime entities (read/write/executable) have to be granted explicitly to software parts Assignment of BSW software modules to partitions => Detailed concept and partition are are available in following svn: https://svn-bmw03-pais.eu.panasonic.com/BMW03_CCU_DOC/Common/WorkSpace/ENG.03_SystemArchitecturalDesign/ENG.03_FS_SafetyConcept/ENG.03_AdditionalDocuments/CCU-6246 FFI SW ASIL Partitioning/ Step to be followed: Follow the memory region defined as Trusted and non-Trusted application based on ASIL Configure the memory regions and allocate the corresponding core MPU Assign Memory regions to OS application based on ASIL Place the code and data in the respective memory location using linker definition Addtional information : QM BSW partitions should run only in user mode to avoid changes to hardware registers If QM module needs supervisor mode access is to be analyzed in FFI and additional mechanism to be implemented ( Example Register protection ) Non-trusted OS-Applications run in non-privileged mode are automatically monitored by the OS If possible use dedicated API provided by OS to call QM drivers access privileged peripheral registers Safe state: Prtitioning should consider the aspect of terminating/restarting applications or the entire CCU in case if CCU application detect a failure. If the application reactions are different for different application error then multiple partition should be available in each core regardless of ASIL level Partitions are defined to be error containment regions
