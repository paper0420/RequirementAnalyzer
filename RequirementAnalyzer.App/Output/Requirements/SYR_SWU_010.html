  • 7.4.1.20 SYR_SWU_010: update main-application
  • 7.4.1.20.1 Description
  • 7.4.1.20.1.0-1 Brief description:
  • 7.4.1.20.1.0-1.0-1 The bootloader will receive UDS service requests by Ethernet/CAN related to software-update. The executed protocol will request: - Identification of device - Deletion of application memory (config data / application program) - Verification of data to be flashed (validity/authenticity) - Flashing of new application - Verification of written data - Start of new application The data which will be flashed can be located in “system CPU” itself OR in any external attached component (DSP, PIC, SPI-Flash of QCA7005) Depending of meta-data, the bootloader will distribute the data to addressed target. If flashing was completed successfully, the bootloader will request start of “main application” to “boot manager” The bootloader is able to process UDS-flash-update procedure. The bootloader will store logging data in “config data” memory for tracing while flash-process is ongoing. (This is independent of additional DLT (Diagnosis Log and Trace) activities, which are not issue of this document) In case of flash-failure the bootloader will stay active until affected, valid application was flashed successfully. In this case the bootloader will execute limited network-powermanagement to prevent standby-current issues. The external busses are allowed to enter “sleep-mode”, but a wakeup of any bus will cause re-activation of bootloader to be able to start new flash-try
