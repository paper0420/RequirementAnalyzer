  • 7.2.1.2 SYR_PMG_020: detect and handle problems at startup
  • 7.2.1.2.1 Description
  • 7.2.1.2.1.0-1 Brief description:
  • 7.2.1.2.1.0-2 While start up the CCU will execute some self-checks for physical and logical failures. In case of detection, the CCU will reduce its functionality or stop all functionality. This UC will cause and handle “ZOMBI” and “DEAD” mode. This UC will start if HW-reset of System-CPU was released and stops if Main-Application becomes working nominally. In case of problems while booting, the “emergency application” is executed. This is original “main application” with these limitations: 1. No write access to any NVM/FLASH 2. No charging via ISO15118 PnC 3. No ISO15118 VAS processing/routing 4. No diagnosis session other than “standard session” If the “main application” is physically corrupted, its intended that the “DEAD”-Counter will count the crashes and the CCU will enter “DEAD condition” sooner or later. If the “main application” itself fails the signature-check, the “bootloader” will be started to enable update (=fix) of “main application”. If the “bootloader” fails signature-check, too, the CCU will be disabled finally. The “signature check” is RSA based hash-checking. While operation, the “main application” will do permanently self-check for consistency, this this fails, the “consistency fail” event will be generated. If (TBD) functionality of main-application was processed successfully, the “everything is fine” signal will be generated to reset “DEAD” and “ZOMBI” counters.
  • 7.2.1.2.1.0-3 Preconditions: CCU is booting (caused by wakeup exor reset)
  • 7.2.1.2.1.0-4 Trigger: no trigger -> atomic sequence
  • 7.2.1.2.1.0-5 Input data: content of flash-memory
  • 7.2.1.2.1.0-6 Description of behaviour: check content of flash memory for validity/authencity / consistency
  • 7.2.1.2.1.0-7 Timing Requirements: overall boot process shall not be > 150ms (this include checking)
  • 7.2.1.2.1.0-8 Output data: qualification of memory content
