  • 7.4.2.14 SYR_SUP_130: Update data/software via XCP
  • 7.4.2.14.1 Description
  • 7.4.2.14.1.0-1 Brief description:
  • 7.4.2.14.1.0-2 This is master Use-Case for XCP update. In general all contexts of data for update is processed outside the CCU. This SW-Update will just access the memory on physical level. By this and update of external components (e.g. PIC / DSP) is not possible. There is some update possible via ETKA (https://www.etas.com/en/products/etk.php?langS=true&), but since the ETKA accessed the SOC via its JTAG/NEXUS interface, this is NOT considered in this document, even the communication between ETKA and INCA is done via XCP. The flash-update vi XCP is handled by supplied “XFF”-software which will be REMOVED in production software, because XCP-Softwareupdate is requested while development, only.
  • 7.4.2.14.1.0-3 Preconditions: none
  • 7.4.2.14.1.0-4 Trigger: XCP communication ongoing / flash shall be updated
  • 7.4.2.14.1.0-5 Input data: data to be flashed
  • 7.4.2.14.1.0-6 Description of behaviour: received data will be flashed into CCU's flash-memory
  • 7.4.2.14.1.0-7 Timing Requirements: none
  • 7.4.2.14.1.0-8 Output data: none
  • 7.4.2.14.1.0-9 Postconditions: data was changed in flash-memory
  • 7.4.2.14.1.0-10 Descriptions of exceptions: none
  • 7.4.2.14.1.0-11 Dependencies and interactions: none

  • 7.4.2.15 SYR_SUP_140: start XFF - obsolete
  • 7.4.2.15.1 Description
  • 7.4.2.15.1.0-1 Brief description:
  • 7.4.2.15.1.0-2 After the System-CPU starts up after “reset”, this UC will check if normal boot procedure will be executed or the XFF bootloader will be started. The conditions to starts the XFF are: a) Not valid boot-strap-loader (=Bootmanager) is available b) Too many (TBD) consecutive “resets” happened without successful start-up of CCU c) XFF operation is requested by XCP; this request will be generated by XCP-Stack of application .
  • 7.4.2.15.1.0-3 Preconditions: CCU is booting
  • 7.4.2.15.1.0-4 Trigger: restart of CCU (reset)
  • 7.4.2.15.1.0-5 Input data: request to start SW_update via XFF
  • 7.4.2.15.1.0-6 Description of behaviour: on request, the XFF will be started and executed
  • 7.4.2.15.1.0-7 Timing Requirements: none
  • 7.4.2.15.1.0-8 Output data: none
  • 7.4.2.15.1.0-9 Postconditions: XFF is running
  • 7.4.2.15.1.0-10 Descriptions of exceptions: The XFF was used in both G08 and I20 during development phase, but then removed from the production sw release.
