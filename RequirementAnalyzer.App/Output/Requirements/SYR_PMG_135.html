SYR_PMG_135


  • 7.2.1.12 SYR_PMG_135: execute emergency shutdown
  • 7.2.1.12.1 Description
  • 7.2.1.12.1.0-1 Brief description:
  • 7.2.1.12.1.0-2 In case of critical situation, the CCU can be shut down in “emergency mode”. This mode will stop all follow-ups and enter sleep-mode as fast as possible. An emergency shutdown cannot be interrupted by wake up event. A wake up event will be stalled and processed (if valid) after CCU entered sleep-mode. The emergency shutdown will ensure that persistent data (NVM) will be consistent and the hardware is configured for sleep-mode correctly. It’s admissible to drop the operating system and disable all interrupts while “emergency shutdown”. The triggers for “emergency shutdown” are: - Undervoltage (<6V >500ms) - Fail of consistency check (CRC/Signature of flash memory under execution) - Request from functional-safety component to enter “SAFE MODE” (-> see TBD) - Detection of major hardware fail (not safety-relevant, but impacting the nominal functions) - Shutdown request by external device via CAN/DIAG request An “emergency shutdown” will NOT increment ZOMBIE/DEAD counter! If there is some request from diagnosis-tester to “reset” the CCU, the “emergency shutdown” will be executed BUT after shutdown, the CCU will restart automatically without applied wakeup event.
  • 7.2.1.12.1.0-3 Preconditions: CCU running nominal
  • 7.2.1.12.1.0-4 Trigger: indication of "emergency situation"
  • 7.2.1.12.1.0-5 Input data: physical value / internal processing result
  • 7.2.1.12.1.0-6 Description of behaviour: CCU will derate its functionality to prevent any kind of damage/impacte to CCU/environment this damage includes virtual damage, too.
